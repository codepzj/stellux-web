# 评论功能设计

## 背景

当前文章详情页已经有“跳转到评论”的交互入口，但页面内还没有真正的评论区。
仓库里也已经存在服务端 `Comment` 模型，说明评论功能不是从零开始，而是需要补齐接口、前台展示和两级回复的完整链路。

## 目标

- 在文章详情页下方新增评论区
- 支持评论列表、发表评论、回复楼中楼
- 仅支持两级评论
- `parent_id` 为空表示一级评论
- `parent_id` 有值表示二级评论
- 回复入口只能挂到一级评论上，不允许继续回复二级评论形成第三层

## 非目标

- 不做无限层级评论树
- 不做评论编辑
- 不做评论删除
- 不做评论审核后台
- 不做通知系统
- 不做点赞、踩、收藏等互动

## 当前状态与约束

- 前台是 Next.js 文章页
- 侧边栏已经有“跳转到评论”的按钮，默认锚点是 `#comment`
- 当前后端只看到 `Comment` 数据模型，还没有对应的 handler/service/repository 实现
- 现有 `Comment` 模型里有这些字段：
  - `post_id`
  - `parent_id`
  - `nickname`
  - `avatar`
  - `email`
  - `website`
  - `content`
  - `device_id`
  - `user_agent`

## 关键设计决策

### 1. 评论列表采用“平铺数据，前端组树”

后端返回某篇文章下的全部评论平铺列表，前端根据 `parent_id` 组装成一级/二级结构。

原因：

- 后端实现更简单
- 前端已经掌握文章页面上下文，组两层树成本低
- 两级结构固定，不需要复杂递归

### 2. 顶层评论和二级评论共用同一套提交接口

提交时只需要决定 `parent_id` 是否存在：

- 一级评论：`parent_id = null` 或不传
- 二级评论：`parent_id = 一级评论 id`

### 3. 只允许回复一级评论

二级评论下不再展示“回复”按钮。

这样可以严格保证结构永远只有两层，避免交互和数据结构继续膨胀。

## 数据契约

### 评论记录

前端使用的评论记录建议统一为：

- `id`
- `post_id`
- `parent_id`
- `nickname`
- `avatar`
- `email`
- `website`
- `content`
- `created_at`
- `updated_at`

### 列表接口

建议接口语义：

- `GET /comment/list?post_id=<post_id>`
- 返回某篇文章下所有未删除评论，按时间排序

### 创建接口

建议接口语义：

- `POST /comment/create`
- 请求体：
  - `post_id`
  - `parent_id` 可空
  - `nickname`
  - `email`
  - `website` 可空
  - `content`
- 服务端补充：
  - `device_id`
  - `user_agent`
  - `avatar` 可由默认头像策略生成或保持为空由前端兜底

## 后端模型调整点

当前 `Comment` 模型里的 `ParentId` 是非空 `ObjectID`，这和“顶层评论 parent_id 为空”不一致。

需要在后端契约里明确一件事：

- 顶层评论必须能表达“没有 parent”
- 实现上建议把 `parent_id` 改成可空字段，或者在 DTO / 序列化层做显式 nullable 处理

如果不做这个调整，顶层评论会被迫落成零值 ObjectID，这会和业务语义冲突。

## 前端组件结构

### `CommentSection`

评论区总入口，职责：

- 拉取评论数据
- 维护当前回复目标
- 维护提交中状态
- 渲染标题、空态、列表和表单
- 提供 `id="comment"` 作为锚点

### `CommentList`

职责：

- 接收评论平铺数组
- 组装一级 / 二级评论结构
- 渲染一级评论和其下的二级评论

### `CommentItem`

职责：

- 渲染单条评论的头像、昵称、邮箱/网址、时间、内容
- 一级评论显示“回复”按钮
- 二级评论不显示回复按钮

### `CommentForm`

职责：

- 发表评论
- 回复模式下显示“回复某人”
- 校验昵称、邮箱、内容
- 提交后清空输入并退出回复态

## 交互流程

### 浏览评论

1. 进入文章详情页
2. 页面加载时请求文章评论
3. 评论区展示一级评论和其二级回复
4. 没有评论时展示空态文案

### 发表评论

1. 用户在底部表单填写昵称、邮箱、内容
2. 点击提交
3. 前端发送创建请求
4. 成功后刷新列表
5. 清空表单

### 回复评论

1. 用户点击某条一级评论的“回复”
2. 表单切换到回复模式
3. 表单顶部提示当前回复对象
4. 提交时自动带上对应一级评论的 `parent_id`
5. 成功后退出回复模式

## 校验规则

- 昵称必填
- 邮箱必填
- 内容必填
- 邮箱格式校验
- `parent_id` 不允许手动输入，只能由 UI 状态控制

## 错误与空态

### 空态

- 无评论时显示“暂无评论，来抢沙发”
- 空态仍保留表单，允许用户直接发第一条评论

### 加载失败

- 评论列表请求失败时显示错误提示和重试按钮
- 不影响文章正文渲染

### 提交失败

- 保留用户已输入内容
- 提示错误原因
- 不自动切换回复目标

## 页面接入点

在文章详情页正文后插入评论区：

- `app/(app)/blog/[alias]/page.tsx`

并保持原有“跳转到评论”按钮不变：

- `components/SideTool/scroll-to-comment.tsx`

只要页面底部存在 `#comment`，现有按钮就能正常工作。
